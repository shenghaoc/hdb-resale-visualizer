# Technical Design

## Architecture
The app remains a static Vite/React application served from precomputed artifacts in `public/data/`. Core domain data must continue to be generated by `scripts/sync-data.ts`; the browser must not fetch from data.gov.sg, OneMap, or any other domain for core HDB functionality.

## Search Model
Introduce an explicit product concept: location scope.

Location scope is true when at least one of these is true:

1. `filters.town` is set.
2. `filters.search.trim()` is non-empty.
3. `filters.selectedAddressKey` is set.
4. A geographic search intent resolves from the search query.

The search text remains location-oriented. It should match block, street, town-like text, MRT station intent, or coordinates. It should not become a general keyword filter for arbitrary attributes such as flat model or flat type, because those already have structured controls.

## Form Layout
`FilterPanel` should present:

1. Core criteria: location search, town, flat type, budget range, remaining lease, MRT distance.
2. Advanced refinements: flat model, floor area range, transaction window.

This keeps the current data contract intact while making the buying workflow clearer.

## Results Visibility
`ResultsPane` should receive a boolean named around scope, not town. The current `hasTownFilter` concept is too narrow and misleading.

Expected behavior:

1. No location scope: empty guidance state.
2. Location scope with no matches: no-match state.
3. Location scope with matches: sortable result list.
4. Selecting a result keeps the results workflow open and swaps in the detail drawer.

## Map Layer Policy
Map layers are split into context layers and decision layers.

Context layers:

1. OneMap base map.
2. MRT lines, stations, and station exits.
3. **Radius Context**: 1km and 2km concentric circles centered on the selected block geocode.

Decision layers:

1. HDB clusters.
2. HDB unclustered block points.
3. Selected block highlight and label.

Visibility rules:

1. Context layers (Map/MRT) are visible by default.
2. Radius Context is visible only when `selectedAddressKey` is present.
3. HDB clusters and unclustered block points are hidden until a location scope exists.
4. HDB clusters and unclustered block points use already-filtered `blocks` passed from `App`.
5. Selected highlight remains independent so URL-selected blocks can still be found.
6. When search is used as the scope, the map should fit to the filtered block bounds via an `autoFitKey`.

## Interaction Policy
1. First meaningful map interaction dismisses the header card but leaves a summon control.
2. Background map interaction closes or hides blocking panels.
3. Feature interaction keeps the active results flow open and selects the clicked block.
4. Double click zoom remains custom and aggressive.
5. Mobile bottom tabs remain fixed and immutable.
6. **Geolocate Control**: implement using standard MapLibre control but styled to match the app's action button design.

## Deterministic Comparison Pipeline
Current artifacts already support price, lease, MRT, transaction count, recency, and trends. Next pipeline work should add static amenity comparisons in this order:

1. Primary schools from MOE general school information, geocoded and cached.
2. Hawker centres from NEA GeoJSON.
3. Supermarkets from SFA GeoJSON.
4. Parks from NParks GeoJSON.
5. Percentile/rank artifacts computed per town and flat type.

Every generated comparison must include source metadata, generated timestamp, and enough raw metric context to avoid black-box scoring.

## Data Sharing Logic
**Formatted Summary**: implement a utility that converts the `rows` array in `ShortlistDrawer` into a Markdown table.
Fields to include: Address, Median Price, Target Fit, Remaining Lease, MRT Proximity, and Amenity Summary (Schools/Hawkers).
